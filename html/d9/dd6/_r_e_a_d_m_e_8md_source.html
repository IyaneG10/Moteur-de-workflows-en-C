1 # PRESENTION DU PROJET
2 
3 L’objectif du projet réside dans la conception et le développement d’un moteur de workflows simplifié pour l’obtention de la convention de stage d’un étudiant passe par différentes activités.
4 Dans cet exemple, nous avons 7 activités différentes et 6 transitions (spécifiés dans un fichier xml correspondant à un processus de demande de stage). Chaque activité est réalisée par un performer qui correspond au login d’un utilisateur connecté. Le parsing des fichiers XML se fait par la méthode SAX avec la bibliothèque libXML2.
5 L'éxecution du code se fait e trois parties:
6 
7 `serveur`: programme principal qui gère à la fois les clients connectés et le processus administrateur
8 
9 `client`: Lancement de thread de gestion des clients connecté au serveur en TCP
10 
11 `administrateur`: Le processus système administrateur communique avec le processus principale via des file de messages IPC en analysant les arguments passés en paramètre.
12 
13 # Contenu du projet
14 Le projet est constitué de trois bibliothèques et des programmes serveur et administrateur. Chacune de ces entités est développée dans un répertoire propre. Le projet s’organise donc autourdes six répertoires suivants :
15 
16 `Socket` - pour les sources permettant de générer la bibliothèque C libsck qui contiendra toutes les méthodes liées au réseau
17 
18 `Threads` - pour la bibliothèque C libthrd qui contiendra toutes les méthodes permettant d’encapsuler l’utilisation des threads.
19 
20 `IPC` - pour la bibliothèque C libipc
21 
22 `Workflow engine` - pour le programme serveur et les modules de traitement des commandes client et administrateur
23 
24 `Workflows admin` - pour le programme d’administration du serveur
25 
26 `Models` - pour les différents modèles de processus existants
27 
28 Un fichier `Makefile` aglobal pour tout le code et des makefiles scpécifiques à chaque réêrtoire

42 # 2°) EXECUTION DU PROGRAMME COTE SERVEUR
43 
44 Se placer sur le répertoire `Workflow_engine` et executer:
45 
46  `./main -p <numPort> -u <fichier> `
47 
48 ou
49 
50  ./main --port <numPort> --users file <fichier>`
51 
52 Ces options permettent de scpécifier éventuellemnt un numéro de port différent de celui par défaut (port 4000) et/ou un autre fichier lors du démarrage du serveur
53 
54 # 3°) UTILISATION DU PROGRAMME COTE CLIENT
55 
56 Se connecter au serveur en TCP/IP via l'utilitaire netcac (nc)
57 
58  `nc localhost <numPort>`
59 
60 A ce moment vous devrez suivre les étapes indiquées par le serveur et lui donner votre login et mot de passe
61 
62 Une fois connecté, vous pouvez intéragir avec le serveur via les commandes suivantes:
63 
64 `aide` : pour connaître les commandes disponibles
65 
66 `ToDo` : pour passer en mode gestion de la To-Do List, i.e., gérer les activités assignées à l’utilisateur
67 
68 `connected users` : pour connaître tous les utilisateurs connectés
69 
70 `ls processes` : pour lister l’identifiant et l’état de tous les processus.
71 
72 `ls processes -ETAT ` : pour ne lister que les processus dans cet état
73 
74 `ls process <id-process>` : pour récupérer toutes les informations associées à un processus particulier
75 
76 `create process <file>` : pour créer une nouvelle instance de processus
77 
78 `exit` : pour quitter l’application et se déconnecter
79 
80 ### Dans le mode ToDo
81 Lorsque qu’un utilisateur est rentré dans le mode ToDo, il peut utiliser les commandes suivantes :
82 
83 `ls` : pour lister toutes les activités assignées à l’utilisateur (récupération du nom et de l’identifiant).
84 
85 `ls <id-activity>`: pour récupérer en plus l’état de l’activité, les entrées et les sorties
86 
87 `valider` : pour valider une activité. Lors de la validation, le serveur demandera à l’utilisateur préciser le numéro du processus concerne puis l'activité à valider , et enfin de donner une nouvelle valeur pour chaque variable résultat (output).
88 
89 `exit` : pour quitter le mode ToDo pour revenir au mode initial
90
